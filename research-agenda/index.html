Research Agenda

#

Truth emerges more readily from error than from confusion.

Francis Bacon

#

I'm not a researcher but I keep a research agenda to give a direction to my career. This agenda guides my self-directed learning and help me filter and organize information.

#

My research agenda is based on the belief that we do not have a real software engineering discipline yet. In particular, I think we need to move away from the focus on the formal representation of programs and towards the design of conceptual solutions to real-world problems.

#

This page is a working document of these ideas and tools.

#

Conceptual Design of Software

#

I don't believe that production code is a good medium to think about the design of software systems. The performance requirements and characteristics of the platforms the software runs on will always lead to some accidental complexities. We should be able to think about our systems in a different medium. Because we do not have this medium yet, we have to go back to "pencil and paper". We need more powerful tools.

#

However, before developing those tools, we need to change the way we think about software and software organizations. We need more powerful ideas.

#

First, we should pay more attention to the problems we're trying to solve. We need to get used to thinking functionally and not in terms of well-known solutions. In "The Problem With User Stories", I illustrate some of the issues of not spending enough time analyzing the problem.

#

We cannot completely describe the problem at the outset of the project. The design process is fundamentally iterative and requires a back and forth between the problem domain and the solution domain. This is why software methodologies that require complete functional specifications before starting the design do not work.

#

However, english prose, like production code, is not a good medium to design software solutions. I believe that one important step to improve software engineering is to focus on the conceptual design of software.

#

Complex Systems

#

Historically the industry has been focusing on algorithms and programs. I believe that we also need to look at our software as a complex system and use the methods and tools to deal with this kind of system.

#

However, the software process deals with many artifacts, not just the code. We need to see the design problem and the conceptual design as complex systems as well. The software organization is a complex system too (see below).

#

Science of Design

#

We're still designing software systems the way Babylonians were doing mathematics—with a collection of best practices and design principles. We're still missing the theories that would allow us to scale our artifacts to higher levels of complexity and make the design process more repeatable.

#

The science of engineering or science of design is a relatively new field that looks at methods to make the design process more rationale.

#

There will always be a part of design that requires creativity and ingenuity. However, we need better thinking tools to control and enhance this creativity.

#

Modeling

#

We too rarely build runnable models of our software systems. Those models are a much better medium than production code to explore the design space.

#

By going one step further we can automatically generate the production code from the model as they did for the Squeak virtual machine.

#

Organization

#

Despite the fact that most of the cost of software projects is generated after the software was released, the industry is still focused on short-term cost savings.

#

A software organization is a complex system. The tools of coupling and modularity apply to organizations as well when designing them. The systems thinking discipline provides a set of ideas and tools to help us steer organizations in the desired direction.
